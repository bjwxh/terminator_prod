## Bug 1 — CRITICAL (FIXED)
**`isinstance` check masks API failures, allowing "healthy" sync with empty data**

**File:** `monitor.py` — `_sync_broker_data` ~L627–664 *(at time of discovery)*

**Problem:**
```python
# Step 1: Both 200s → broker marked CONNECTED (before data is parsed)
if resp_acc.status_code == 200 and resp_ord.status_code == 200:
    with self._data_lock:
        self.broker_connected = True      # ← marked healthy BEFORE parsing
        self.heartbeat_failures = 0

# Step 2: isinstance fires → ord_data silently becomes []
ord_data = resp_ord.json()
if not isinstance(ord_data, list):
    self.logger.warning(...)              # ← only a warning
    ord_data = []                         # ← silent reset

# Step 3: sync "succeeds" with empty data
self.live_combined_portfolio.trades = []  # ← wipes all live trades
self.working_orders = []
```
The Schwab API can return a dict instead of a list on paginated responses or auth edge cases. The guard caught this but silently discarded the data instead of failing.

**Fix:** Parse and validate `ord_data` **before** marking `broker_connected = True`. Raise on non-list so the except block increments `heartbeat_failures` and the UI shows an alert.

**Depends on:** nothing
**Depended on by:** Bug 2 (corrupt session), Bug 3 (empty bootstrap source)

## Bug 2 — HIGH (FIXED)
**`live_combined_portfolio.trades` replaced every 5 seconds instead of merged**

**File:** `monitor.py` — `_sync_broker_data` ~L659 *(at time of discovery)*

**Problem:**
```python
self.live_combined_portfolio.trades = broker_trades  # full replace every 5s
```
A single 5-second window with an empty API response (e.g. transient network, rate limit) wipes the entire live trade history. The corrupted portfolio is then saved to the session file.

**Fix:** Merge by `order_id` instead of replacing:
```python
existing_ids = {t.order_id for t in self.live_combined_portfolio.trades if t.order_id}
new_trades = [t for t in broker_trades if t.order_id and t.order_id not in existing_ids]
if new_trades:
    self.live_combined_portfolio.trades.extend(new_trades)
self.live_combined_portfolio.cash = sum(t.credit for t in self.live_combined_portfolio.trades)
```

**Depends on:** Bug 1 (Bug 1 is the primary trigger of empty `broker_trades`)
**Depended on by:** Bug 3 (session saved with empty trades feeds corrupt bootstrap)

## Bug 3 — HIGH (FIXED)
**Bootstrap captures `live_combined_portfolio.trades` before broker sync has ever run**

**File:** `monitor.py` — `_monitoring_loop` ~L534 *(at time of discovery)*

**Problem:**
All `asyncio.TaskGroup` tasks start concurrently. `_monitoring_loop` runs its synchronous setup — including the `broker_trades` capture — before any `await` yields to `_broker_sync_loop`. The first capture is always `[]`.

```python
with self._data_lock:
    broker_trades = list(self.live_combined_portfolio.trades)  # ← always [] on first start
```
If the session was saved during a Bug 2 erasure window, `restore_monitor` also restores empty trades, compounding the issue.

**Fix:** Perform an explicit broker sync **before** reading `broker_trades`:
```python
self.logger.info("Performing initial broker sync before bootstrap...")
await self._sync_broker_data()

with self._data_lock:
    broker_trades = list(self.live_combined_portfolio.trades)  # ← now populated
```

**Depends on:** Bug 2 (corrupt session as a secondary path)
**Depended on by:** nothing (terminal in the cascade)

## Bug 4 — CRITICAL (PENDING)
**Bootstrap silently crashes: chart-padding code uses stale tz-naive `dt_series`**

**File:** `monitor.py` — `_run_historical_simulation` ~L2235–2262

**Problem:**
`dt_series` is captured from `data['datetime']` *before* timezone conversion. The conversion modifies `data['datetime']` in-place but does not update `dt_series`. Subsequent code then uses the stale tz-naive `dt_series` against tz-aware group keys:

```python
# L2235: captured while column is still tz-naive
dt_series = pd.to_datetime(data['datetime'])

# L2237: column is converted in-place — dt_series NOT updated
data['datetime'] = dt_series.dt.tz_localize('America/Chicago')

# L2244: groups keyed by tz-AWARE timestamps
groups = data.groupby('datetime')

# L2255: TypeError — tz-naive key against tz-aware group index
first_snap = groups.get_group(dt_series.iloc[0])   # ← CRASH

# L2262: TypeError — comparing tz-aware curr_pad with tz-naive first_avail_ts
while curr_pad < first_avail_ts:                    # ← CRASH
```

This branch fires whenever `first_avail_ts.time() > 08:30` — i.e. on every mid-day deployment where the DB has no data before the restart time. The exception propagates to `_monitoring_loop` L562, is caught, and logged. **The bootstrap silently exits, leaving all portfolios empty** (they were reset at L542–547 but never re-populated).

**Cascade after crash:**
- `live_combined_portfolio` is repopulated by the next broker sync (correct)
- `combined_portfolio` and all sub-strategy portfolios stay empty
- First `_monitor_step` fires: all 3 strategies have `has_traded_today = False` → all call `_check_entry` simultaneously → `net_trades()` collapses 3 × 4-leg model ICs into **one 12-leg net trade** at arbitrary model-predicted strikes
- `session_history` starts collecting at restart time → **charts start at 11:11am**

**Fix:**
Read `first_avail_ts` from the already-converted column, not from the stale series:

```python
# Replace L2255–2257:
first_avail_ts = data['datetime'].iloc[0]             # tz-aware Chicago Timestamp
first_snap = groups.get_group(first_avail_ts)         # key matches group index
first_spx = self.estimate_spx_price(first_snap) if not data.empty else None
```
The comparison `while curr_pad < first_avail_ts` then becomes tz-aware vs. tz-aware.

**Depends on:** nothing (independent crash)
**Depended on by:** Bug 5 and Bug 6 are masked by this bug (bootstrap never reaches their code)

## Bug 5 — HIGH (PENDING)
**BROKER IC mapping silently discards ICs filled before `trade_start_time`**

**File:** `monitor.py` — `_run_historical_simulation` ~L2320–2340

**Problem:**
When mapping an untagged BROKER `IRON_CONDOR` trade to a sub-strategy, the code requires the current DB snapshot time (`ts`) to have already passed the sub-strategy's `trade_start_time`:

```python
for s_id in available_strat_ids:
    s_obj = self.sub_strategies[s_id]
    if not s_obj.has_traded_today and ts.time() >= s_obj.trade_start_time:
        target_sid = s_id
        break
# If no strategy passes: falls through to —
else:
    self.combined_portfolio.add_trade(sim_trade)  # combined only, no sub-strategy
    # sub-strategy portfolio stays empty
```

`ts` is the DB snapshot being processed at the moment the IC is popped from `pending_live_trades`, not the IC's actual fill time. If an IC is filled 1–5 minutes before the strategy's designated start (which is normal, e.g. fill at 09:14 for a strategy starting 09:15), the snapshot at 09:14 has `ts.time() < trade_start_time` and no sub-strategy is matched.

**After-effect:** `_reconcile_combined_simulation()` rebuilds `combined_portfolio.positions` from sub-strategy portfolios — which are empty for the unmatched strategy. Combined positions are wiped for that strategy → direct mismatch with `live_combined_portfolio`.

**Fix:**
Remove the start-time gate. ICs in `pending_live_trades` are already sorted chronologically and `available_strat_ids` is already sorted by `trade_start_time`. Sequential assignment is unambiguous:

```python
# Replace:
if not s_obj.has_traded_today and ts.time() >= s_obj.trade_start_time:

# With:
if not s_obj.has_traded_today:
```

**Depends on:** Bug 4 must be fixed first (otherwise bootstrap crashes before reaching this code)
**Depended on by:** nothing

## Bug 6 — MEDIUM (PENDING)
**Fuzzy leg-signature deduplication silently drops valid ICs from strategies with identical strikes**

**File:** `monitor.py` — `_run_historical_simulation` ~L2188–2196

**Problem:**
```python
leg_sig = tuple(sorted([(l.symbol, l.strike, l.side, l.quantity) for l in t.legs]))
ts_fuzzy = t.timestamp.replace(second=0, microsecond=0)  # truncate to minute
sig_key = (ts_fuzzy, leg_sig)

if sig_key in seen_signatures:
    continue  # ← second IC silently dropped
seen_signatures.add(sig_key)
```

When two sub-strategies share the same `init_s_delta` and `init_l_delta` (same delta targets), they produce ICs at the same strikes. If both ICs fill within the same calendar minute, they share the same `(minute, leg_sig)` key. The second IC is dropped, leaving one sub-strategy with no opening trade in the sim.

The fuzzy dedup was intended to prevent replaying split-execution fills of a single order. That case is already handled correctly by the order-ID check at L2182–2186 (same order → same `order_id` → deduped). The fuzzy check is redundant for that case and harmful for the multi-strategy case.

**Fix:**
Remove the fuzzy signature block and its initialisation:

```python
# Remove: seen_signatures = set() at L2178
# Remove: the entire leg_sig / ts_fuzzy / sig_key / seen_signatures block at L2188–2197

# Keep only Order ID deduplication (L2182–2186) — it is sufficient.
# Two valid ICs from different strategies always have different order_ids.
```

**Depends on:** Bug 4 must be fixed first (otherwise bootstrap crashes before reaching this code)
**Depended on by:** nothing

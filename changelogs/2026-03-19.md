# Changelog — 2026-03-19

All changes stem from the audit comparing `terminator_prod` against the original standalone `spt_v4` desktop app.
Full audit details: [`AUDIT.md`](../AUDIT.md)

---

## Critical Fixes

### Trade Execution — `order_queue` Was Never Drained
**File:** `server/core/monitor.py`

The original port left `monitor.order_queue` populated by the reconciliation loop but nothing ever consumed it. Trades were silently queued and dropped — the app would never execute a live trade.

**Fix (developer):** Implemented `run_order_execution_loop()` — an async background task that drains the queue every second and calls `execute_net_trade()` for each pending trade. The loop is started via `tg.create_task()` inside `_run()`.

---

### WebSocket Protocol Mismatch — Dashboard Showed No Live Data
**File:** `server/api/ws.py`

During the trade execution refactor, the WS broadcaster and the JS client were left out of sync. `broadcast_state()` sent the raw state dict with no type field, but `app.js` was updated to only call `updateUI()` when it received `{"type": "state_update"}`. Since the type field was never present, `updateUI()` was never called and the entire dashboard displayed stale zeroes.

**Fix (Claude):** Wrapped the state broadcast in the expected envelope:
```python
# Before
await manager.broadcast(state)
# After
await manager.broadcast({"type": "state_update", "state": state})
```

## Greek Values Showing 0.0 (Race Condition Fixes)

### Bug 6a + 6b — Snap Fallback Not Written Back to Cache
**File:** `server/core/monitor.py` — `_update_live_portfolio()`

`_greek_cache` is empty for the first ~30 seconds after startup (the cache is only populated by `_monitor_step` which runs every 30s). During that window, `_broker_sync_loop` fires every 5s and falls back to `_last_snap` for Greeks — but the result was never saved back to `_greek_cache`. Every broker sync re-hit the same fallback, and any partial snap failure left positions with `delta=0.0, theta=0.0`.

**Fix (developer):** After a successful snap fallback lookup, the result is now written back: `self._greek_cache[k] = (delta, theta)`, progressively warming the cache.

---

### Bug 6c — Cache Sentinel Confusion (`or` Condition)
**File:** `server/core/monitor.py` — `_update_live_portfolio()`

The condition `if (delta == 0.0 or theta == 0.0)` used `0.0` as both the "cache miss" sentinel and a legitimate Greek value. Deep OTM positions with a real delta near zero would always trigger unnecessary snap re-lookups, potentially overwriting valid cached values.

**Fix (developer):** Changed to `if k not in self._greek_cache` — snap fallback only triggers on a true cache miss, never on legitimately zero values.

---

### Bug 6d — WS Broadcaster Read Positions Without Lock
**File:** `server/api/ws.py` — `broadcast_state()`

`broadcast_state()` read `live_p.positions` and `sim_p.positions` outside `_data_lock`. Since `_update_live_portfolio()` clears `positions = []` then appends, a read between those two operations would compute `total_delta = 0.0`. In pure asyncio this was unlikely to fire, but the lock discipline was broken.

**Fix (developer):** Moved the entire state dict construction inside a single `with monitor._data_lock:` block. Positions lists are snapshotted atomically under the lock.

## Sound Alerts — Moved to Web UI

### New: `_broadcast_alert()` Helper + Web UI Handler
**Files:** `server/core/monitor.py`, `server/static/app.js`

`spt_v4` used `afplay` (macOS) for audio alerts. This was removed in `terminator_prod` since the server runs headless on a VM. The responsibility was moved to the browser.

**Implementation (developer):**
- Added `_broadcast_alert(level, title, message)` helper to `monitor.py` that fires a `{"type": "alert", "level": "chime"|"error", ...}` WebSocket event.
- Alert events are triggered at the same points `spt_v4` called `afplay`: when a new trade signal is queued (chime) and on broker sync errors (error).
- Added `handleAlert()` to `app.js` which plays the appropriate audio file (`/static/chime.mp3` or `/static/error.mp3`) and triggers a browser push notification.

### Fix: Trade Execution Alert Used Wrong WS Type
**File:** `server/core/monitor.py` — `run_order_execution_loop()`

The auto-execution loop broadcast `{"type": "trade_executed", ...}` which had no handler in `app.js` — the chime would never play on a live trade execution.

**Fix (Claude):** Replaced the ad-hoc broadcast with a call to `_broadcast_alert("chime", ...)` so trade executions use the same alert pathway as everything else.

## Mobile Responsiveness

`style.css` had zero `@media` queries. The layout was desktop-only and broken in multiple ways on phones and tablets.

**All fixes implemented by developer in `server/static/style.css`:**

| Issue | Fix |
|-------|-----|
| Status bar items overflow, buttons clipped off-screen | Added `flex-wrap: wrap` to `#status-bar`; mobile breakpoint reduces padding and gives buttons full width |
| Tab nav clips last 1–2 tabs out of reach | `overflow-x: auto; white-space: nowrap` on `.tab-nav` at ≤ 600px |
| Tables overflow with `overflow: hidden` — content invisible | Changed `.table-container` to `overflow-x: auto` |
| `2rem` container padding wastes 17% of screen width | Reduced to `1rem` at ≤ 600px |
| Metrics grid collapses to 1 column on phones | `repeat(2, 1fr)` at ≤ 600px |
| `.btn-small` touch target ~26×18px (below 44px minimum) | Increased padding on mobile |
| 3 modal footer buttons too cramped in one row | `flex-direction: column-reverse` at ≤ 480px |
| `background-attachment: fixed` causes iOS Safari scroll stutter | Changed to `scroll` at ≤ 600px |
| `.trades-grid` class used in HTML with no CSS definition | Added explicit `grid-template-columns: 1fr 1fr` rule with mobile stack |

# Changelog: April 8, 2026

## New "Chase" Functionality for Working Orders

Implemented a "Chase" feature to allow one-click price improvement for working orders directly from the dashboard.

### **Highlights**
*   **One-Click Price Improvement**: Added a green **"Chase"** button to the "Action" column in the Working Orders table.
*   **Intelligent Rounding**: Implemented logic to automatically calculate the "less aggressive" price improvement (nearest 5 cents):
    *   **Credit Spreads**: Rounds **UP** (e.g., $5.27 → $5.30) to capture more credit.
    *   **Debit Spreads**: Rounds **DOWN** (e.g., $5.27 → $5.25) to pay less debit.
*   **Confirmation Workflow**: Integrated with the system's confirmation modal to prevent accidental price changes.
*   **Efficient Order Replacement**: Utilized the Schwab API's `replace_order` (PUT) mechanism to update prices without losing position in the queue (where supported by the broker), re-sending the full order specification as required.

## Fixes & Improvements (v1.1)
*   **Precision Safety**: Migrated rounding logic to use `decimal.Decimal` to prevent floating-point inaccuracies (e.g., preventing $5.25 being rounded to $5.20 due to IEEE 754 precision).
*   **Robust Strategy Tracking**: Improved order-to-strategy mapping to retain existing associations if the broker does not immediately return a new order ID during replacement.
*   **Snappy UI**: Added immediate table refresh after a successful "Chase" action, ensuring the user sees the updated price without waiting for the next polling cycle.
*   **UI Polish**: Corrected a typo in the confirmation modal description.

## DB Health Monitoring (v1.2)
*   **Real-time Data Watchdog**: Added a background task that monitors the database for data lag every 1 minute during market hours.
*   **Intelligent Status Rendering**: New "DB" indicator in the status bar with three states:
    *   **Healthy**: Data is less than 2 minutes old (Green).
    *   **Lag (Xm)**: Data is stale, showing the actual age in minutes (Red).
    *   **Sleep**: Silenced during off-market hours or weekends (Gray).
*   **Automated Alerts**: 
    *   **Log Tab**: Issues a system-wide warning log when lag is detected.
    *   **Audio**: Triggers a warning sound (`error.mp3`) on the dashboard upon transition to "Lag" state.
    *   **Email**: Sends a critical warning email to configured recipients when the DB transitions to a lagging state.

## DB Monitoring Fixes (v1.2.1)
*   **Empty DB Resilience**: Fixed a bug where a completely empty database during market hours failed to trigger alerts. It now treats "zero records" as a critical lag event.
*   **One-Shot Alert Logic**: Converted `should_alert` into an edge-triggering flag that resets immediately after the first WebSocket broadcast, preventing redundant audio alerts on page refreshes.
*   **Enhanced Logging**: Added proper warning logs when falling back to default email recipients due to missing configuration.

### **Technical Details**
*   **Backend (`monitor.py`)**: Added `chase_order(order_id)` method to handle core rounding logic and API interaction.
*   **API (`routes.py`)**: Added `POST /api/orders/{order_id}/chase` endpoint.
*   **Frontend (`app.js`, `index.html`)**: Added DB status element and logic to process real-time health updates via WebSocket.
*   **WebSocket (`ws.py`)**: Included DB health state in the 500ms broadcast heartbeat.
*   **Styling (`style.css`)**: Added `.btn-success` utility class for the new button.
